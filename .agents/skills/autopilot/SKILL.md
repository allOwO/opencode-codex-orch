当此 skill 被激活时，以下规则立即生效。

> **重要**：Autopilot 仅覆盖**冗余确认节点**（鹦鹉式复述、"可以吗"式暂停）。
> 它**不覆盖** mandatory process skills（如头脑风暴发现、计划审查、质量门控）。
> 如果某个 workflow skill 的步骤是为了**收集新信息或做质量校验**，autopilot 不跳过它。

## 第一原则：不做鹦鹉

这是最高优先级规则，无条件执行。

当用户意图明确时：
- ❌ **禁止**复述用户指令再问"可以吗" / "如果你同意"
- ❌ **禁止**展示执行步骤再问"要开始吗" / "准备好了吗"
- ❌ **禁止**列出计划后等待批准
- ❌ **禁止**说"我理解你想要…，对吗？"
- ✅ 直接执行
- ✅ 如果需要做假设，做完假设直接执行，在最终汇报中列出假设

**判断标准：**

```
如果 用户意图明确 且 操作可逆或低风险：
    → 直接执行

如果 用户意图模糊 或 操作不可逆且高风险：
    → 只问真正不确定的部分，不复述已知信息
```

**反例（绝对禁止）：**
- 用户说"翻译成中文" → 你说"如果你同意，我翻译成中文" ← 废话
- 用户说"加个日志" → 你说"我计划在第 42 行添加 log，可以吗？" ← 直接加
- 用户说"重命名变量" → 你说"步骤：1. 找到 2. 替换 3. 测试。开始？" ← 直接做
- 用户说"可以" / "好" / "行" → 你再问一次确认 ← 直接下一步

## 第二原则：跳过冗余确认，保留有价值的流程

仅跳过**纯确认性质的暂停**，不跳过**信息收集或质量校验步骤**。

### 可以跳过的（冗余确认）
- "准备好了吗？" / "要开始吗？" 式的过渡确认
- 方案选择（2-3 个方案让你选）→ 自动采用推荐方案并说明理由
- Spec 文件写完后的审查等待 → 写完直接进入下一步
- 执行方式选择 → 优先 subagent-driven（有支持时），否则 inline
- 合并方式选择 → 默认创建新分支但不合并
- 清理确认 → 保留 worktree 不自动清理

### 不能跳过的（mandatory process）
- 头脑风暴/发现阶段的**信息收集**（如澄清需求、探索 codebase）
- 计划审查（Momus review 等质量门控）
- Task 间的**错误检测检查点**（出错时必须暂停）
- 最终 review 的**实际校验步骤**（跑测试、读代码）

## 第三原则：安全兜底

即使在 Autopilot 模式下，以下情况 **必须停止并询问**：

1. 测试反复失败（≥ 3 次同一原因）
2. 缺少关键依赖且无法自动解决
3. 需求存在根本性矛盾（不是模糊，是矛盾）
4. 破坏性操作（删库、force push、覆盖生产数据）
5. 安全敏感操作（API key、secrets、认证凭据）
6. 在 main/master 分支上直接修改（应自动创建分支）

## 完成时汇报格式

任务完成后，输出一份简洁汇报，不要求确认，不问"还需要什么"：

```
📋 Autopilot 汇报
- 需求: [一句话]
- 方案: [选了什么方案、为什么]
- 假设: [自行做出的假设列表，没有则写"无"]
- 变更: [创建/修改的文件列表]
- 测试: [通过/失败情况]
- 分支: [分支名，如适用]
- 需关注: [需要人工检查的事项，没有则写"无"]
```
